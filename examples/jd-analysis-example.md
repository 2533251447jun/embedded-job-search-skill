# JD Analysis Example

All JD entries below are **Example JD / Example Analysis** material. They are teaching examples, not real company postings.

## 示例 JD 1：STM32 / MCU 初级固件岗

### JD 摘要

岗位要求 C 语言、STM32、GPIO/UART/I2C/SPI/ADC、CubeMX/HAL，能配合硬件调试；加分项包括 FreeRTOS、RS485、示波器。

### 岗位族判断

- 主岗位族：STM32/MCU 固件。
- 次岗位族：硬件调试 / 联调。
- 岗位难度：初级，偏外设实现和调试。

### 必备技能

- `fact`: JD 写明 C、STM32、基础外设、CubeMX/HAL。
- `inference`: 面试会追问外设初始化、中断/轮询、日志和调试。

### 加分项

- FreeRTOS：需要任务、队列、信号量或 demo 证据。
- RS485：需要串口日志、协议帧或硬件接线证据。
- 示波器：需要真实波形使用记录。

### 隐含要求

- 编码比例：60%
- 联调比例：25%
- 测试比例：15%
- 隐含要求：能解释项目边界，不把课程项目说成企业项目。

### 初级候选人匹配项

- `fact`: 有 STM32 + UART/I2C/ADC 课程项目。
- `inference`: 可投初级固件岗位，但需准备外设追问。

### 短板与 needs_proof

- `needs_proof`: FreeRTOS 任务调度。
- `needs_proof`: 示波器波形。
- `needs_proof`: RS485/Modbus 项目经验。

### 投递建议

谨慎可投。简历主打 STM32 基础外设、串口日志和调试记录；FreeRTOS/RS485 写学习中或补充 demo。

### 高风险面试追问

- UART 中断接收如何避免丢字节？
- I2C 无 ACK 怎么排查？
- ADC 采样时间和参考电压如何影响读数？
- 你是真做过 FreeRTOS，还是只看过教程？

### 不建议投递的情况

- JD 要求独立负责量产固件、复杂 RTOS 架构或 Linux 驱动。

### 不允许包装的方向

- 不得写“精通 STM32 底层架构”。
- 不得写“主导量产固件开发”。

### 需要补证据的任务

- 补 1 个 FreeRTOS 基础任务 demo。
- 补 1 份外设调试日志。
- 补 1 个串口或 I2C 异常复现记录。

## 示例 JD 2：嵌入式测试工程师

### JD 摘要

岗位要求嵌入式功能测试、测试用例、串口日志、缺陷复现、回归测试；加分项包括 Python 脚本、CANoe/CANalyzer、HIL/SIL。

### 岗位族判断

- 主岗位族：嵌入式测试。
- 次岗位族：车载通信测试，如果 JD 明确 CAN/UDS/HIL。
- 岗位难度：初级到初中级，重视测试记录。

### 必备技能

- `fact`: JD 写明测试用例、日志、缺陷复现、回归测试。
- `inference`: 面试会要求候选人讲清 Bug 生命周期。

### 加分项

- Python：需要真实脚本或自动化片段。
- CANoe/CANalyzer：需要真实工具使用截图或日志。
- HIL/SIL：无平台证据只能写了解。

### 隐含要求

- 编码比例：10%
- 测试比例：70%
- 联调比例：20%
- 隐含要求：能把开发经验降级为测试证据，不把测试岗写成开发岗。

### 初级候选人匹配项

- `fact`: 有串口日志和功能验证记录。
- `inference`: 可准备“异常复现 + 复测”案例。

### 短板与 needs_proof

- `needs_proof`: 完整测试报告。
- `needs_proof`: 自动化测试脚本。
- `needs_proof`: HIL/SIL 平台经验。

### 投递建议

可投。简历应突出测试用例、日志分析、缺陷复现、回归验证，不要把自己包装成固件开发负责人。

### 高风险面试追问

- 一个缺陷从发现到关闭如何记录？
- 你如何判断修复有效？
- 串口日志中哪些字段证明问题复现？
- 没有正式测试报告时如何说明？

### 不建议投递的情况

- JD 明确要求多年车载 HIL 平台经验或完整自动化测试体系建设。

### 不允许包装的方向

- 不得写“负责完整质量体系”。
- 不得写“主导 HIL/SIL 平台测试”。

### 需要补证据的任务

- 整理 3 条测试用例。
- 补一份缺陷复现记录。
- 准备一段串口日志分析说明。

## 示例 JD 3：工业控制 / RS485 / Modbus

### JD 摘要

岗位要求 STM32、RS485、Modbus RTU、CRC、UART、现场联调意识；产品方向为数据采集终端或工业设备通信模块。

### 岗位族判断

- 主岗位族：工业控制 / 数据采集。
- 次岗位族：STM32/MCU 固件、硬件联调。
- 岗位难度：初级偏实践，若要求现场经验则升级为初中级。

### 必备技能

- `fact`: JD 写明 RS485、Modbus、UART、CRC。
- `inference`: 面试会追问寄存器地址、异常码、超时和抗干扰。

### 加分项

- 现场联调：需要真实现场或实验记录。
- PLC：无实际连接证据时只能写了解。
- 长稳测试：需要运行记录或测试报告。

### 隐含要求

- 编码比例：45%
- 联调比例：35%
- 测试比例：20%
- 隐含要求：能够解释协议帧，不把 UART 收发写成完整协议栈。

### 初级候选人匹配项

- `fact`: 有 STM32 UART 基础。
- `inference`: 可通过 Modbus RTU demo 补强。

### 短板与 needs_proof

- `needs_proof`: RS485 实物接线。
- `needs_proof`: Modbus CRC 代码。
- `needs_proof`: 多设备轮询和超时机制。
- `needs_proof`: 现场抗干扰。

### 投递建议

谨慎可投。建议先补一个 STM32 + RS485 + Modbus RTU 基础 demo，保留课程/个人项目边界。

### 高风险面试追问

- Modbus RTU 帧格式是什么？
- CRC16 如何计算？
- 寄存器地址和功能码如何对应？
- 从站无响应如何处理？
- RS485 A/B 接反怎么排查？

### 不建议投递的情况

- JD 要求独立负责 PLC 现场交付、工业网关量产或多年现场抗干扰经验。

### 不允许包装的方向

- 不得写“主导工业协议栈设计”。
- 不得写“负责客户现场调试和量产交付”。

### 需要补证据的任务

- 至少一个真实项目或 demo。
- 至少一个真实调试案例。
- 至少一个协议分析案例：帧、CRC、超时、异常码。
